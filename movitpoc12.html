Moovit WebView / deep-link report — companion PoC

This page is a truthful runtime witness for researchers and triage. It does not simulate reading Moovit’s internal account database from JavaScript.

What this file honestly supports (if opened inside Moovit via your deep link)
  • Attacker-controlled HTTPS content can load in the app’s WebView when the url= parameter points here — you can see this page’s URL and timestamp in the JSON output.
  • JavaScript in this origin can read normal WebView-exposed data (URL, query string, navigator, screen/viewport, cookie string length, presence of window.mobile).
  • Optional: HTTP(S) requests from this page (e.g. to a researcher-controlled endpoint) prove network egress from the WebView context — use your own test URL; this file does not embed a secret webhook.
What this file does not prove by itself
  • It does not extract real full name, email, phone, payment card, or tickets from Moovit private storage. Those are not available to arbitrary page JS unless the app exposes them (bridge, injected data, or URL).
  • Intent redirection (non-http(s) schemes handled via Intent.parseUri) is a separate reproduction: use an appropriate intent: / custom-scheme link or navigation as in your app analysis — not demonstrated by the JSON below unless you trigger that path.
  • If your report shows profile/payment screens or webhook captures, that evidence comes from those steps and screenshots/video, not from this HTML reading the DB.
Report field mapping (honest)
The JSON includes profile.full_name, email, mobile, payment_card, tickets only if you pass them as query parameters on this page’s URL (for structured demos). Otherwise they are null. This makes the data lineage explicit in the output under profileSource.
